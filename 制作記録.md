## 機能一覧

1. **ロールプレイタイマー**:本番と同じ「ロールプレイ15分+口頭試問5分」の構成。実時間(壁時計)基準で動くため、スマホの画面ロック等で処理が間引かれても残り時間は狂わない。残り1分・切替・終了で音の合図(画面を見ずに練習できる。ただし画面ロック中は音が鳴らないため、画面は点けたまま伏せて置く)。
2. **振り返りチェックリスト**:態度・展開・自己評価の3観点。自己評価に加えて「他者評価」欄(相談者役や録画を見た人が、声のトーン・表情・話しやすさを評価)を用意し、自己認識とのズレを検出できるようにした。記録は端末のブラウザ内(localStorage)に保存され、日時付きで履歴に残る。
3. **傾向分析**:記録が2回以上たまると、ルールベースで「修正すべきこと/改善できたこと/気を付けること」を自動生成。加えて「AIに深掘り分析を依頼」ボタンで、全履歴を分析用プロンプトに整形してクリップボードにコピーでき、Claude等のAIチャットに貼るとより踏み込んだ講評が受けられる。
4. **論述 時間配分シート**:受験団体(協議会/JCDA)を切り替え可能。協議会側は公式サイトで公開されている第29〜31回の実物過去問を読んで作成した、実際の設問構成・配点に基づく配分(下記参照)。
5. **バックアップ/復元**:履歴をテキストとしてコピー・貼り付けで復元できる(端末変更時やデータ消失時の対策)。

## 実際に起きたトラブルと対処(再現時に必ず当たるポイント)

制作中に実際に発生した問題とその解決方法。別PCで同じ手順を踏むと、同じところで詰まる可能性が高い。

### 1. `git push` が `fatal: 'origin' does not appear to be a git repository` で失敗
GitHub Desktopでリポジトリを新規作成した直後、ローカルのGitリポジトリにリモート(origin)が設定されていなかった。原因は、GitHub Desktop上での操作とコマンドラインのgitが別々にリポジトリを認識していたこと。
**対処**:
```
git remote add origin https://github.com/<ユーザー名>/<リポジトリ名>.git
git fetch origin
```
さらに、Web UI経由でファイルをアップロードした際の履歴(README作成など)とローカルの履歴が「関係のない履歴」として衝突したため、
```
git merge origin/main --allow-unrelated-histories -X ours
git push -u origin main
```
で、ローカルの最新ファイルを優先する形で統合した。**別PCで新規にリポジトリを作る場合、GitHub Desktopでの作成とコマンドライン操作を混在させると同じ状態になりやすいので、どちらか一方の操作方法に統一するのが望ましい。**

### 2. GitHub Pagesの初回ビルドが失敗する(公開URLが更新されない)
pushが成功しても、GitHub Pagesのビルド(`pages build and deployment`)が1回目は`failure`になり、公開URLの内容が更新されないことがあった。原因はJekyllの自動処理との相性と推測される(明確なエラーメッセージは確認できず)。
**対処**:リポジトリのルートに空の `.nojekyll` ファイルを追加してpushすることで、Jekyllのビルド処理をスキップさせ、静的HTMLがそのまま配信されるようにした。
```
touch .nojekyll
git add .nojekyll && git commit -m "Add .nojekyll" && git push
```
**別PCで新規にPagesを設定する場合、最初から`.nojekyll`を置いておけばこの問題は避けられる。**

### 3. ページ更新後、ブラウザ・確認ツールに古い内容が表示される
pushしてPagesのビルドが成功した直後でも、確認に使ったツール(WebFetch)やブラウザのキャッシュにより、しばらく旧バージョンが表示され続けた。
**対処**:1〜3分待って再取得する、またはURLの末尾に `?v=2` のようなダミーのクエリパラメータを付けてキャッシュを回避する。**お父様がツールを開いても更新が反映されていないように見える場合、まず「ブラウザの再読み込み(スーパーリロード)」を試してもらうこと。**

### 4. クリップボードへの自動コピーが失敗する(「コピーできませんでした」)
「AIに深掘り分析を依頼」ボタンで、ブラウザ環境によっては `navigator.clipboard.writeText` が使えず、コピーが失敗した(実際にお父様の環境で発生)。
**対処**:コピー処理を3段フォールバック化した。
1. Clipboard API(`navigator.clipboard.writeText`)
2. 失敗時は旧方式の `document.execCommand('copy')`
3. それも失敗したら、画面にテキストエリアを表示して手動で全選択・コピーしてもらう
**この修正は `index.html` に組み込み済みのため、別PCで一からツールを作る場合も同じ3段フォールバックを最初から入れておくとよい。**

## 別のPCで同じ状態を使う/編集する方法

### そのまま使うだけなら(お父様が使う分には)
何もする必要なし。上記の公開URLをブックマークすれば、どのPC・スマホからでも同じツールが開ける。データ(練習履歴)は開いた端末ごとに別々に保存される点に注意。

### 別のPCで編集を続けたい場合
**注意:以下は手順として妥当だが、別PCでの動作は未検証。** 特にGit認証(手順1)は、このPCではGitHub Desktopのサインイン情報が既に資格情報マネージャーに残っていたためpushが通っており、別PCでは認証エラーで一度は止まる可能性が高い。

1. そのPCにGitを導入し、GitHub Desktopでサインインするか、Personal Access Tokenを発行して認証を通す
2. `git clone https://github.com/wison-blip/career-consultant_practice.git`(既に`.nojekyll`もリポジトリに含まれているので、Pages設定さえ済ませればビルド失敗は起きないはず)
3. `index.html` を編集
4. `git add index.html && git commit -m "変更内容" && git push`
5. 数分でGitHub Pages(公開URL)に自動反映される。反映が確認できない場合は上記トラブル3のキャッシュを疑う

### Claudeに続きを頼む場合
このファイル(`制作記録.md`)と、ローカルの `cc-shiken-practice.html` をそのPCの同じフォルダ構成に置いた上で、「このツールを引き継いで修正してほしい」と伝えれば、Claudeは経緯を踏まえて作業できる。

### 別の受験者向けに似たツールを1から作る場合
Claudeにツール本体(HTMLファイル)は作らせられるが、**GitHubへの公開はClaude単独では完結しない**。この環境には`gh`コマンド(GitHub CLI)が入っておらず、新規リポジトリの作成とGitHub Pagesの有効化はブラウザでの手作業が必要だった。実際の分担は以下の通り。

**Claudeに伝える情報**:
- 受験団体(協議会/JCDA)
- 前回の弱点(論述の時間切れ/構成ミス、面接の態度/展開/自己評価のどこで失点したか)
- 受験回数・残り期間

**利用者が手作業で行う必要がある部分(今回の実績)**:
1. GitHubで新規リポジトリを作成(**リポジトリ名にスペースを入れない**。今回`career consultant-practice`と入力しかけて`career-consultant-practice`に修正した)
2. ファイルをアップロードする際、ファイル名を**`index.html`**にする(既定のファイル名のままだとGitHub Pagesがトップページとして認識しない)
3. リポジトリのSettings → Pages → Source を `Deploy from a branch`、Branchを`main`/`/(root)`に設定してSave
4. 上記トラブル2の通り、`.nojekyll`を最初から一緒にアップロードしておくとビルド失敗を避けられる

この4点はClaudeが指示はできても、ブラウザ操作そのものは利用者が行う必要がある(GitHub認証がこの環境に無いため)。

## 運用上の注意点

- 練習記録はブラウザのlocalStorageに保存されるため、**端末やブラウザを変えると引き継がれない**。定期的に「履歴をバックアップ(コピー)」でテキスト保存しておくこと。
- ツールを更新するたびに、GitHubへのpushを忘れると公開URL側が古いままになる(Artifact版とPages版の二重管理になっているため)。
- 受験団体・設問構成・配点は年度によって変わりうるため、本番が近づいたら協議会公式サイトの最新過去問で再確認するのが望ましい。

## この記録の更新ルール

このファイルは「今回のセットアップで実際に何が起きたか」の記録であり、一度書いて終わりにすると次第に実態とズレていく。**ツール(`index.html`)に機能追加・修正を行った回、または新しいトラブルに遭遇して対処した回は、そのつどこのファイルにも追記すること。** 具体的には:

- 機能を追加/変更したら → 「機能一覧」の該当項目を書き換える
- 新しい不具合とその対処が発生したら → 「実際に起きたトラブルと対処」に章番号を追加して記録する(番号は起きた順で追番)
- 試験の実物情報(設問構成・配点・過去問の内容)を追加調査したら → 「調査で判明した試験の実物情報」を更新し、参照した過去問の回・出典URLを明記する
- ツール本体を更新した際は、Artifact・ローカル・GitHubの3箇所へのコピー/pushが完了してから、最後にこのファイルを更新する(このファイルだけ更新して実体が古いままにならないように)

このファイル自体の更新履歴(いつ何を直したか)は、Gitのコミット履歴(`git log -- README.md` 相当)で追える設計にしているため、ファイル内に更新日時の欄は設けていない。
